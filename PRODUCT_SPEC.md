# AllOfSoccer Product Spec (Living Document)

이 문서는 앞으로 사용자(프로덕트 오너)가 말하는 스펙을 계속 누적 관리하는 기준 문서다.
최종 업데이트: 2026-02-12

## 1) 제품 목표
- 축구 회장들이 주말 경기 매칭을 더 쉽게 등록/성사할 수 있는 플랫폼
- 성사된 매치에 용병이 쉽게 지원할 수 있는 구조
- 회장도 용병 모집 글을 빠르게 올릴 수 있는 구조

## 2) 핵심 사용자
- 회장/운영진: 경기 개설, 상대팀 매칭, 용병 모집
- 일반 사용자(용병): 기존 매치에 용병 지원

## 3) 핵심 기능 방향
### 3.1 팀 매치 탐색/필터
- 지역별 필터
- 날짜 필터
- 경기 종류 필터
- 매칭 상태 필터 (모집중/완료 등)
- 필터는 iOS 화면 로컬 필터와 서버 쿼리 결과가 동일한 의미 체계를 가져야 한다.
- 지역 카테고리 필터는 운영 카테고리(서울북부/서울남부/경기북부/경기남부/인천부천/기타지역) 기준으로 동작한다.

참고: 위 필터는 현재 코드/화면 의도상 이미 존재하며, 완성도와 연결성(실데이터, UX)을 계속 높인다.

### 3.2 지역 분류(운영 카테고리)
- 서울북부 평일축구
- 서울북부 주말축구
- 서울남부 평일축구
- 서울남부 주말축구
- 경기북부 평일축구
- 경기북부 주말축구
- 경기남부 평일축구
- 경기남부 주말축구
- 인천부천 평일축구
- 인천부천 주말축구
- 기타지역 평일축구
- 기타지역 주말축구

추가 카테고리(운영 필요 시):
- K5, K6, K7 팀 매치 신청

### 3.3 매치/용병 플로우
- 회장: 매치 등록 -> 상대팀 매칭
- 회장: 매치 또는 별도 모집글로 용병 모집
- 용병: 기존 매치/모집글에 지원
- 회장 작성 생산성: "이전 글 불러오기"를 통해 기존 글을 복사하고 날짜/시간/장소만 수정해 재등록 가능해야 한다.

### 3.4 카페 글 대체용 표준 입력 항목 (기본값)
아래 항목은 팀 매칭 글 작성 시 기본으로 제공되어야 한다.
- 경기 형태: 축구/풋살
- 팀 정보: 팀 이름, 연령대, 팀 소개
- 팀 실력: 실력 레벨, 선출 포함 여부/인원
- 지역/구장: 지역 + 구장명
- 일정: 매치 날짜 + 시간
- 비용: 구장비 총액 + 상대팀 부담 비용
- 연락처: 전화번호 (문자/전화 유도)
- 경기 방식: 11:11, 6:6 등
- 코멘트: 매너 요청, 약속/취소 정책, 기타 안내

## 4) 커뮤니케이션 정책 (중요)
- 앱 내 메시지는 지양한다.
- 회장 간 연결은 전화번호 공개를 기본으로 한다.
- 매치 게시글에 연락처(전화번호)를 명확히 노출하여 문자/전화로 바로 연결되게 유도한다.

## 10) 팀/멤버십 구조 (신규)
- 사용자는 팀을 생성할 수 있다.
- 한 팀에는 여러 사용자가 "팀원"으로 참여할 수 있다.
- 팀 운영자(회장/운영진)는 팀원 초대/승인/권한 관리가 가능해야 한다.
- 팀 단위로 매치 이력, 평점, 운영 정보를 누적 관리한다.

## 11) 매칭 성사 후 상호 평가 (신규)
- 트리거: 두 팀 간 매칭이 "성사/완료" 상태가 되면 평가 가능 상태로 전환한다.
- 평가 대상: 상대 팀 단위
- 평가 항목:
- 매너 점수 (예: 1~5)
- 실력 점수 (예: 1~5)
- 선택 코멘트
- 원칙:
- 상호 합의된 매칭에서만 평가 가능
- 매칭 당 1회 평가(수정 가능 기간은 정책으로 제한)
- 팀 평점은 누적 평균/최근 N경기 추세를 함께 제공

## 12) 실력 불일치 완화 장치 (신규)
- 선출 포함 여부/인원 노출을 강화한다.
- 팀 평점(실력/매너) 기반으로 상대팀이 매칭 리스크를 사전에 판단할 수 있게 한다.
- 장기적으로는 "예상 체감 난이도" 또는 "실력 편차 경고"를 제공하는 것을 목표로 한다.

## 13) 제품 미션 (재확인)
- 네이버/다음 카페 중심의 PC 기반 매칭 운영을 모바일 중심으로 이전한다.
- 게시글 작성/복사/재등록/연락/성사/평가까지 앱에서 닫힌 흐름으로 제공한다.

## 9) 데이터/운영 원칙
- iOS 앱의 샘플 데이터 의존도를 줄이고, 로컬 서버 seed 데이터로 실제 API 연동 기준 검증을 우선한다.
- 로컬 서버 seed에는 최소한 아래 상태를 포함한다.
- 팀매치: `recruiting`, `full` 또는 `completed`
- 용병모집: `recruiting`, `closed`
- 마이페이지(내가 쓴 글/내 활동)도 동일한 서버 데이터 기반으로 동작해야 한다.

## 5) Non-Goal
- 앱 내 채팅/메신저 중심 구조는 현재 우선순위가 아니다.

## 6) 운영 원칙
- 스펙은 이 문서에 계속 누적 업데이트한다.
- 구현 시 문서와 코드 간 차이가 생기면 문서를 먼저 최신화한 뒤 반영한다.

## 7) 팀 구성 (Squad)
- 기획(Product): 요구사항 정의, 우선순위/로드맵 관리, 릴리즈 스코프 결정
- 디자인(Design): IA/플로우, 화면 설계, 컴포넌트/디자인 시스템 관리
- iOS: iOS 앱 기능 구현, 배포, 안정화
- Server: API/DB/인증/운영 백엔드 구현 및 성능/보안 관리
- QA: 테스트 전략, 회귀/통합/E2E 검증, 품질 게이트 운영

## 8) Android 확장 계획
- 추후 Android 팀을 추가 구성한다.
- Android 앱은 iOS와 기능 동등성(Feature Parity)을 목표로 한다.
- 우선순위: 핵심 플로우(매치 탐색/필터/등록/용병 지원·모집) 동등 구현 -> 세부 UX 정합 -> 운영 자동화.
- 정책: 신규 기능은 iOS/Android/Server/QA가 같은 스펙 기준으로 동시 관리한다.
