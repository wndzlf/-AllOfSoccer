# iOS + 기획 + 디자인 합동 완성도 점검

기준일: 2026-02-12  
목표: 회장 중심 매치 성사율을 높이고, 용병 지원/모집 흐름을 단순화하며, UI 일관성을 강화한다.

## 1) 현재 강점
- 지역/날짜/경기종류/상태 기반 필터 구조가 이미 존재한다.
- 팀 매치/용병 모집/마이페이지/등록 플로우가 앱 내 기본 골격으로 연결되어 있다.
- 최근 P0 이슈를 중심으로 API 연동 및 상세 화면 구현이 진행됐다.

## 2) 핵심 부족점 (합동 관점)
### 2.1 기획 관점
- 핵심 정책인 "연락처 기반 외부 커뮤니케이션"이 화면/검증/데이터 반영에서 일관되게 강제되지 않았다.
- 등록 플로우의 필수 입력/선택 입력 구분이 화면마다 다르고 안내가 약하다.

### 2.2 디자인 관점
- 동일한 의미의 색상/radius/폰트가 파일마다 하드코딩되어 시각 일관성이 흔들린다.
- 카드, 배지, 버튼의 스타일 기준이 화면마다 미세하게 달라 브랜드 인지가 약해진다.
- 빈 상태/에러 상태 표현 방식이 화면별로 통일되지 않았다.

### 2.3 iOS 구현 관점
- 일부 화면에 TODO가 남아 있어 사용자 여정이 끊길 수 있다.
- 등록 시 소개글 placeholder와 실입력 데이터가 혼동될 여지가 있었다.
- 연락처가 실제 서버 전송 데이터에 명시적으로 남지 않는 경우가 있었다.

## 3) 이번 반영 사항 (코드)
- TeamMatch 목록 -> 상세 화면 네비게이션 TODO 제거
- 팀 모집 등록 시 연락처 필수/형식/동의 검증 추가
- 소개글 placeholder 방어 로직 추가
- 연락처를 매치 설명(description)에 포함해 외부 연락 유도 정책 반영

## 4) 다음 우선순위 (Sprint 제안)
### P1. 디자인 시스템 통합
- 공통 토큰(색/간격/타이포/컴포넌트)을 정의하고 하드코딩 상수를 단계적으로 대체
- 대상 우선 화면: TeamMatch, MercenaryMatch, MyPage 리스트 계열

### P1. 기획 UX 고도화
- 등록 플로우에서 필수값 라벨/검증 메시지/동의 문구를 통합 규격으로 정리
- 연락처 노출 정책을 상세 화면에도 동일하게 드러내기

### P2. iOS 품질 강화
- 주요 플로우 단위 테스트 보강(등록 검증, 필터 파라미터 변환, 페이지네이션)
- 빈 상태/에러 상태 컴포넌트 공통화

## 5) 팀별 액션 아이템
### 기획팀
- 연락처 노출 정책을 PRD 필수 요구사항으로 고정
- 필터 우선순위와 지역 카테고리 운영 규칙 확정

### 디자인팀
- 공통 UI 토큰 표준안 작성 (v1)
- 상태 배지/CTA/폼 입력 컴포넌트 가이드 확정

### iOS팀
- 공통 토큰 적용 리팩터링
- 등록/상세/리스트의 상태 UI 일관화

### QA팀
- 연락처 정책 검증 케이스 추가
- 필수값 미입력/잘못된 연락처/동의 미체크 케이스 회귀 자동화
